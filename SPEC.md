# SPEC.md — Recomp · 8 Weeks

### Frontend: single-file app

`public/index.html` — all HTML, CSS, and JavaScript inline. No build step, no bundler, no framework.

**Deployment:** `firebase deploy --only hosting`
**Do not** use `git push` — GitHub Actions workflow has a broken `npm ci` step.

### Backend: Firebase

| Service | Usage |
|---------|-------|
| **Firebase Auth** | Email/password. One registered user. |
| **Firestore** | User state. Single document `users/{uid}`. |
| **Firebase Hosting** | Serves `public/` directory. |
| **Cloud Functions** | `generatePlan` — calls Claude API to build personalised plan. Requires **Blaze plan** (upgrade at Firebase Console). |

### Data flow

```
User action
  → Update localStorage immediately (instant UI)
  → Write to Firestore in background

Page load / first data access
  → Check localStorage first
  → If missing, fetch from Firestore and cache locally
```

App is fully functional offline after first load. Firestore is the cross-device sync layer.

## Firestore schema

All data lives in `users/{uid}` — a flat key/value document.

### Meal completion
```
checked_{dow}_{weekNum}  →  { [mealIndex]: true }
```
- `dow` 0–6 (JS day-of-week, 0=Sun)
- `weekNum` 1–8 (program week, computed from `START`)
- Example: `checked_2_1` = Tuesday week 1

### Daily check-ins
```
daily_checkins  →  { "YYYY-MM-DD": { weight: number, bf: number } }
```
- `weight` in lbs, `bf` as % (e.g. `18.5`)
- Requires confirmation to edit or remove

### Prep checklist
```
prep_sun  →  { [itemId]: true }
prep_thu  →  { [itemId]: true }
```

### Shopping list
```
shop  →  { [itemId]: true }
```

### Plan (generated by wizard)
```
wizardInputs  →  { goal, currentWeight, weightUnit, goalWeight, currentBf,
                   timeline, diet, dietNotes, allergies[], allergyNotes,
                   cookingLevel, trainingDaysPerWeek, trainingDays[],
                   trainingType, equipment[], supplementsInclude,
                   supplements[], supplementNotes }

plan  →  {
  weeklyPlan: {
    "0"–"6": {
      type: "training" | "rest",
      meals: [{ time, name, desc, cal, p, c, f }]
    }
  },
  macroTargets: {
    training: { cal, p, c, f },
    rest:     { cal, p, c, f }
  },
  recipes: [{
    name, cat, prep, cook, serves,
    macros: { cal, p, c, f },
    ingredients: string[],
    steps: string[]
  }],
  prepPlan: {
    sun: { label, items: [{ id, name, note, time }] },
    thu: { label, items: [{ id, name, note, time }] }
  },
  shoppingList: [{ cat, items: [{ id, name, qty }] }],
  supplements: string[],
  notes: string
}

planMeta  →  {
  createdAt: ISO string,
  updatedAt: ISO string,
  version: number (increments on each rebuild),
  timelineWeeks: number,
  startDate: "YYYY-MM-DD",
  endDate: "YYYY-MM-DD",
  schemaVersion: 1
}
```

**One plan per user** — enforced by the Cloud Function (reads existing `planMeta.version`, increments it, overwrites `plan` and `wizardInputs`). Progress data (`daily_checkins`, `checked_*`, `prep_*`, `shop`) is preserved across rebuilds.

## Static data (hardcoded defaults, not in Firestore)

Used when no generated plan exists. Once a plan is generated, `loadUserPlan()` overrides these in memory.

| Variable | Contents |
|----------|---------|
| `MEALS` | 7-day meal plan keyed by DOW (0–6) |
| `MACRO_TARGETS` | `{ training, rest }` calorie+macro targets |
| `RECIPES` | 9 recipe objects |
| `RECIPE_CATS` | `['All', 'Batch Cook', 'Salads & Bowls']` |
| `PREP_DATA` | `{ sun, thu }` prep session objects |
| `SHOP_DATA` | Array of `{ cat, items[] }` shopping groups |
| `MEAL_RECIPE_MAP` | Meal name → RECIPES index (for recipe link icons) |

### Active data variables

After `loadUserPlan()` runs on auth, these override the static constants:

```javascript
let activeMeals          = MEALS;
let activeMacroTargets   = MACRO_TARGETS;
let activeRecipes        = RECIPES;
let activeRecipeCats     = RECIPE_CATS;
let activePrepData       = { sun: PREP_DATA.sun, thu: PREP_DATA.thu };
let activeShopData       = SHOP_DATA;
```

All render functions reference `active*` variables, never the static constants directly.

## Plan Creation Wizard

### Trigger logic

```
onAuthStateChanged(user)
  → routeAfterAuth()
    → loadUserPlan()          // load any existing plan
    → checkIfNeedsWizard()    // returns true if brand-new user with no data
      → true:  showWizard(isFirstTime=true)   // shows "skip" link on step 1
      → false: show main app + renderAll()

Profile menu → "Rebuild My Plan"
  → confirmation dialog
  → confirmRebuild() → showWizard(isFirstTime=false)
```

`checkIfNeedsWizard()` returns `false` (skip wizard) if the user has any of:
- `planMeta` field (completed wizard before)
- `daily_checkins` data (legacy/active user)
- Any `checked_*` keys (active user)

### Steps

| Step | Title | Required fields |
|------|-------|----------------|
| 1 | Goals | goal, currentWeight, goalWeight (currentBf optional, timeline required) |
| 2 | Diet | diet, cookingLevel (allergies + notes optional) |
| 3 | Training | trainingDays (≥1), trainingType (equipment optional) |
| 4 | Supplements | supplementsInclude |
| 5 | Review + Generate | — (shows summary, calls Cloud Function) |

### Generate flow

1. Click "Build My Plan" → `wzGenerate()`
2. Show loading overlay with rotating status messages (3.5s each)
3. Call `firebase.functions().httpsCallable('generatePlan')` with 5-minute timeout
4. On success → `window.location.reload()` (plan loads on next auth state)
5. On error → show error message + "Try again" retry button

## Cloud Function: `generatePlan`

**File:** `functions/index.js`
**Runtime:** Node 20, us-central1, 300s timeout, 512MiB memory
**Requires:** Firebase Blaze plan (upgrade at Firebase Console → Usage & Billing)

**API key setup:**
```bash
firebase functions:secrets:set ANTHROPIC_API_KEY
# or via environment variable in Cloud Functions settings
```

The function reads `process.env.ANTHROPIC_API_KEY`.

**Flow:**
1. Verify Firebase Auth token
2. Validate wizard inputs (goal, currentWeight, goalWeight required)
3. Build structured prompt from wizard inputs
4. Call `claude-opus-4-6` with `max_tokens: 8000`
5. Strip markdown fences if present, parse JSON
6. Validate JSON structure (all 7 days, macroTargets, recipes, shoppingList, prepPlan)
7. Retry once on parse/validation failure
8. Save `wizardInputs`, `plan`, `planMeta` to Firestore with `merge: true`
9. Return `{ success: true }`

**Deployment:**
```bash
firebase deploy --only functions
```

## Open questions / decisions

- Firebase project must be upgraded to **Blaze plan** to deploy Cloud Functions. Free Spark plan does not support them.
- `ANTHROPIC_API_KEY` must be set as a Cloud Functions environment variable (via `firebase functions:secrets:set ANTHROPIC_API_KEY` or Firebase Console → Functions → Environment variables).
- Plan consumption: after `loadUserPlan()`, the active data overrides work for meals, recipes, prep, shop. However, `TRAINING_DAYS` (which days are training vs rest) is still a hardcoded constant. A generated plan that sets different training days will correctly mark days as `type: "training"` in `weeklyPlan`, but `TRAINING_DAYS` (used for day tab styling) won't reflect this. This should be updated in `loadUserPlan()`.
- Should `wizardInputs` be considered sensitive data? It includes body weight and body fat — likely fine in Firestore given one-user context, but worth noting.
- The `planMeta.startDate` is set to the date the plan was generated, not the user's actual program start. These may differ.
